## Melhores Práticas de DevOps & Desenvolvimento

### Filosofia ShiftLeft

**Princípio central**: Identificar e corrigir problemas o mais cedo possível no ciclo de vida do desenvolvimento, movendo teste e segurança para o início do processo em vez de deixá-los para o final.

**Práticas de otimização de fluxo**:
- Fazer os itens se moverem da esquerda para direita mais rapidamente
- Tornar o trabalho visível em todas as etapas
- Limitar o trabalho em andamento (WIP) para reduzir mudança de contexto
- Reduzir o tamanho dos lotes para ciclos de feedback mais rápidos
- Reduzir o número de transferências entre times/funções
- Identificar e elevar continuamente nossas restrições/gargalos
- Eliminar adversidades e desperdícios no fluxo de valor

**Mecanismos de qualidade e feedback**:
- Ver problemas quando eles acontecem
- Criar loops de feedback antecipados em todo lugar
- Aglomerar e resolver problemas para construir novo conhecimento organizacional
- Manter a qualidade mais próxima da fonte
- Permitir a otimização para núcleos de trabalho mais adiante
- Institucionalizar a melhoria do trabalho diário
- Transformar descobertas locais em melhorias globais
- Injetar padrões de resiliência em nosso trabalho diário
- Lideranças reforçam cultura de aprendizado

### Modelo de Cultura Organizacional de Westrum

Em sistemas sociotécnicos complexos, o desempenho depende de como a organização trata informação ruim, risco e falha.

**Princípio-chave**: Cultivar uma cultura livre de culpa

### Práticas de Deploy e Release

**Automação e simplificação**:
- Simplificar e automatizar o maior número possível de etapas
- Empacotar o código de maneiras adequadas para implantação
- Criar imagens ou contêineres de máquina virtual pré-configurados
- Automatizar a implantação e configuração de middleware
- Executar testes de fumaça automaticamente

**Estratégias avançadas de deployment**:
- Dark launching (teste de funcionalidades em produção sem visibilidade do usuário)
- Feature toggles (configuração de funcionalidades em tempo de execução)
- Blue-Green deployment (releases com zero downtime)
- Canary releases (rollout gradual com monitoramento)

### Artigos

#### [Dark Launching](https://martinfowler.com/bliki/DarkLaunching.html)
**Conceito**: Técnica para lançar novos comportamentos de backend chamando-os de usuários existentes sem que os usuários saibam que estão sendo chamados. Usado para avaliar carga adicional e impactos de performance no sistema antes de fazer um anúncio público da nova capacidade.

**Como funciona**: Implementa-se a nova funcionalidade com uma interface de controle (Keystone Interface), integrando-a nos releases de produção mas sem interface de usuário visível. O sistema chama a nova funcionalidade durante operações normais (ex: checkout), mas não mostra os resultados aos usuários. Usa Feature Flags para ligar/desligar facilmente em produção.

**Aplicação prática**: Ideal para testar impacto de performance de novas funcionalidades (ex: motor de recomendações) antes do lançamento público. Permite também rodar implementações antigas e novas em paralelo para comparação de resultados.

---

#### [Feature Toggles (Feature Flags)](https://martinfowler.com/articles/feature-toggles.html)
**Conceito**: Técnica poderosa que permite times modificarem comportamento do sistema sem mudar código. Toggles permitem ativar/desativar funcionalidades em tempo de execução através de configuração.

**Categorias principais**:
- **Release Toggles**: Permitem trunk-based development, mantendo código incompleto latente em produção
- **Experiment Toggles**: Para testes A/B e multivariados, direcionando diferentes usuários para diferentes implementações
- **Ops Toggles**: Controle operacional para degradar ou desabilitar funcionalidades sob carga alta
- **Permissioning Toggles**: Habilitam funcionalidades para conjuntos específicos de usuários (premium, beta, internos)

**Implementação**: Varia de configuração estática (hardcoded) a sistemas distribuídos sofisticados (Zookeeper, etcd, Consul). Toggles devem ser temporários quando possível - adicione tarefas de remoção ao backlog ou estabeleça "datas de expiração".

**Complexidade de testes**: Com múltiplos toggles, surgem explosão combinatória de estados possíveis. Testar configuração que irá para produção + configuração de fallback. Convenção: legacy behavior = Off, new behavior = On.

#### [Blue-Green Deployment](https://martinfowler.com/bliki/BlueGreenDeployment.html)
**Conceito**: Técnica que reduz downtime e risco durante deployment mantendo dois ambientes de produção idênticos (Blue e Green). Um ambiente está ativo enquanto o outro fica idle.

**Como funciona**: 
1. Ambiente Blue está live servindo usuários
2. Nova versão é testada completamente no ambiente Green
3. Quando validado, o roteador direciona todo tráfego para Green
4. Blue fica como ambiente de rollback imediato se necessário
5. Após estabilização de Green, Blue vira staging para próximo release

**Vantagens**: Rollback rápido (apenas trocar roteador de volta), testa procedimento de disaster-recovery a cada release. Ambientes podem ser hardware diferente, VMs diferentes, ou zonas separadas no mesmo OS.

**Desafio de banco de dados**: Separar deployment de mudanças de schema de upgrades de aplicação. Primeiro aplique refatoração de database para suportar ambas versões (antiga e nova), depois faça deploy da nova versão da aplicação.

#### [Canary Release](https://martinfowler.com/bliki/CanaryRelease.html)
**Conceito**: Técnica para reduzir risco de introduzir nova versão de software em produção fazendo rollout gradual para pequeno subconjunto de usuários antes de disponibilizar para todos.

**Como funciona**:
1. Deploy nova versão para subset da infraestrutura sem rotear usuários
2. Quando confiante, rotear pequena porcentagem de usuários (1% - cohort canary)
3. Monitorar métricas de negócio (engajamento, receita) entre cohorts
4. Gradualmente aumentar porcentagem de usuários e servidores
5. Rollback é simples: redirecionar usuários de volta para versão antiga

**Estratégias de seleção de usuários**: Amostra aleatória, usuários internos primeiro, ou baseado em perfil/demografia. Em cenários distribuídos: por região geográfica, por marca, etc.

**Diferença de A/B testing**: Canary release detecta problemas/regressões (minutos a horas). A/B testing testa hipóteses (dias para significância estatística). Evitar conflação das duas técnicas.

**Origem do nome**: Mineradores levavam canários em gaiolas para minas de carvão - se gases tóxicos vazassem, matariam o canário antes dos mineradores. Canary release fornece forma similar de alerta precoce.

#### [Site Reliability Engineering (SRE)](https://www.ibm.com/br-pt/think/topics/site-reliability-engineering)
**Conceito**: Prática de engenharia de software que combina DevOps e operações de TI tradicionais para resolver problemas dos clientes, automatizar tarefas de operações de TI, acelerar entrega de software e minimizar riscos de TI.

**Regra 50/50**: Engenheiros SRE dedicam metade do tempo resolvendo problemas de clientes (gerenciar escalonamentos, responder incidentes) e outra metade automatizando operações de TI (gerenciamento de sistemas, mudanças, resposta a incidentes).

**Princípios fundamentais**:
- Falhas são inevitáveis - identificar causas raízes e prever falhas futuras
- Usar dados de monitoramento e registro para análise preditiva
- Configurar automações para resolver problemas, criando resiliência e redundância
- Reduzir necessidade de tarefas manuais de administração

**Ferramentas e técnicas**:
- Monitoramento em tempo real para correção imediata e antecipação de problemas
- Logging como histórico para análise e melhoria de observabilidade
- Engenharia do caos: causar falhas intencionais para entender impactos e desenvolver planos de mitigação
- Planejamento de capacidade para escalar funções e desenvolver novos aplicativos

**Métricas principais**:
- **SLA** (Service Level Agreement): Termos e condições entre provedor e cliente (ex: 99.999% uptime)
- **Orçamentos de erro**: Nível aceitável de risco de erro alinhado com SLAs (ex: 4min 23s downtime/mês para 99.999%)
- **SLO** (Service Level Objective): Metas de performance acordadas para serviços específicos
- **SLI** (Service Level Indicator): Medições quantitativas (uptime, latência, throughput, taxas de erro)

**Relação com DevOps**: DevOps foca em "O que este software deve fazer?". SRE responde "Como pode ser implementado e mantido para funcionar como esperado?". SRE fornece dados reais de performance para DevOps, equilibrando teoria e prática.

#### [Trunk-Based Development](https://trunkbaseddevelopment.com/)
**Conceito**: Modelo de branching de controle de versão onde desenvolvedores colaboram em código em uma única branch chamada 'trunk' (ou 'main'), resistindo a qualquer pressão para criar outras branches de desenvolvimento de longa duração através de técnicas documentadas.

**Benefícios**: Evita merge hell, não quebra build, habilita Continuous Integration e Continuous Delivery. Quando indivíduos commitam para trunk múltiplas vezes por dia, fica fácil satisfazer requisito central de CI (todos commitam ao menos uma vez a cada 24 horas).

**Para times pequenos**: Commits diretos para trunk após build pré-integração local (compile, unit tests, integration tests).

**Para times escalados**: Short-lived feature branches (produto de único dev workstation, seja solo, pair ou mob programming) com Pull Request workflow. Branches são de curta duração e passam por code review e verificação de CI antes de merge.

**Caveats importantes**:
- Dependendo de tamanho de time e taxa de commits, usar short-lived feature branches para code review
- Times pequenos podem commitar direto para trunk
- Podem existir release branches cortadas do trunk just-in-time, hardened antes do release, e **deletadas** após release
- Alternativamente: releasing from trunk com estratégia "fix forward" para bug fixes
- Dominar técnica de Branch by Abstraction para mudanças longas
- Usar Feature Flags para desenvolvimento diário, permitindo hedge na ordem de releases

**Prova de escala**: Google pratica TBD com 35.000 desenvolvedores e QA automators em um único monorepo trunk.

#### [Chaos Monkey](https://netflix.github.io/chaosmonkey/)
**Conceito**: Ferramenta da Netflix responsável por terminar instâncias aleatoriamente em produção para garantir que engenheiros implementem seus serviços para serem resilientes a falhas de instância.

**Objetivo**: Força as equipes de engenharia a construírem sistemas que podem sobreviver e se recuperar de falhas de componentes individuais. Parte da filosofia de "engenharia do caos" - testar resiliência causando falhas controladas.

**Como funciona**: Chaos Monkey aleatoriamente "mata" instâncias em execução em produção. Se o serviço não é resiliente o suficiente, isso causará interrupção. Se for bem projetado, outros componentes assumem sem impacto para usuários.

**Filosofia Netflix**: Preferem descobrir fraquezas em sistemas durante horas de trabalho quando equipes estão disponíveis para responder, em vez de descobrir às 3h da manhã durante incidente real.

**Integração com Spinnaker**: Configurável via Spinnaker (plataforma de continuous delivery da Netflix) para customizar comportamento por aplicação/serviço.
